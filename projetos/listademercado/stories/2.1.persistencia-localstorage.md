# Story 2.1 — Persistência em localStorage

**Épico:** 2 — Persistência
**Status:** Done
**Owner:** Dev
**Estimativa:** M (≤ 0.5d)

## Story
Como **usuário**, quero que **minha lista sobreviva entre sessões**, para que eu possa **fechar o navegador / o celular descarregar / voltar amanhã** sem perder o que digitei.

## Acceptance Criteria
1. **Toda mutação** (add, toggle, remove, clear) grava o estado no `localStorage` antes de retornar (write-through).
2. Chave fixa: **`listademercado.v1`** (versionada — ver ADR/architecture §9).
3. Ao carregar a página, o estado é restaurado dessa chave.
4. **Falha de leitura/parse não quebra o app** — cai silenciosamente para `{ items: [] }`.
5. **Falha de escrita** (cota cheia, modo privado restrito) não quebra o app — engolida silenciosamente; app continua em memória.
6. Shape inválido no JSON (ex.: `items` não é array) é tratado como ausência → `{ items: [] }`.

## Dev Notes
**Origem:** PRD §6 / Story 2.1, FR5. Architecture.md §5.3, §9 (estratégia + tratamento de erro), §10 ADR-002 (`localStorage` em vez de IndexedDB).

**Contrato (`src/storage.js`):**
```js
load(): { items: Item[] }   // sempre retorna shape válido
save(state): void            // best-effort, sem throw
```

**Por que `localStorage` e não IndexedDB:**
- Volume pequeno (texto + bool por item).
- API síncrona simplifica `mutar → persistir` no `store` (sem propagar `async/await` por toda a base).
- Limite de ~5MB folgado pra dezenas/centenas de itens.

**Por que chave versionada (`v1`):** se v2 mudar o shape do `Item`, podemos ler `v1` e migrar para `v2` sem destruir dados antigos (architecture §9).

**Por que silenciar erros:** o caso de uso é trivial e não-crítico; mensagem assustaria sem agregar valor. Em modo privado strict, app continua funcional como sessão volátil.

**Boot:** `store.js` faz `const state = load()` no top-level do módulo — uma única leitura, sem polling, sem `storage` event (multi-aba não é requisito v1).
