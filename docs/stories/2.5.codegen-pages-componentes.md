# Story 2.5: Codegen — `builder.json` -> arquivos `.tsx` Next.js + Tailwind

## Contexto
Epico 2. Cobre Componente 4 (Codegen) da arquitetura. Materializa FR4, FR5 (legibilidade), NFR3. Determinismo (ADR-2): mesmo schema produz mesmos bytes.

## Descricao
Como no-coder, quero que cada alteracao no canvas/painel de props se materialize em arquivos `.tsx` idiomaticos no projeto, salvos automaticamente, para que o codigo seja real, legivel e exportavel.

## Tarefas tecnicas
1. [ ] Criar `lib/builder/codegen/index.js` (orquestrador): recebe `(projectPath, builderJson, affectedPaths)`, decide quais arquivos re-renderizar (incremental).
2. [ ] Criar `lib/builder/codegen/page.js` que emite `app/<route>/page.tsx` a partir de `pages[i]`. Importa componentes Tailwind idiomaticos. Usa template strings.
3. [ ] Criar `lib/builder/codegen/component.js` com renderers por tipo (Container, Heading, Text, Button, Input, Form, List, Table, Image) — cada um vira JSX. Componentes ficam em arquivos separados (1 componente = 1 arquivo) para reduzir blast radius (mitigacao §8 da arquitetura).
4. [ ] Criar `lib/builder/codegen/header.js` que prepende `// @builder:managed schema=<sha8>` em cada arquivo gerado, calculando sha do fragmento relevante do `builder.json`.
5. [ ] Rodar `prettier` (nova dep) sobre cada arquivo emitido; persistir via `fs.writeFile`. Apos cada patch, atualizar `files{}` no `builder.json` com `{ managed: true, schemaHash }`. Acionado automaticamente em `applyPatch` (Story 2.1).

## Arquivos envolvidos
- `lib/builder/codegen/index.js` - criar (orquestrador incremental)
- `lib/builder/codegen/page.js` - criar
- `lib/builder/codegen/component.js` - criar (renderers por tipo)
- `lib/builder/codegen/header.js` - criar (sha + marker)
- `lib/builder/store.js` - modificar (chamar codegen apos applyPatch)
- `package.json` - modificar (dep `prettier`)
