# Story 1.6: Stop do preview e cleanup de zombies

## Contexto
Story do Epico 1. Fecha Componente 5. Critico: cleanup mal feito vira zombie processes consumindo RAM (cada `next dev` ~300MB). Cobre AC do epico "Fechar aba do Jarvis Hub mata os processos de preview".

## Descricao
Como Diogo, quero garantir que parar o preview ou desligar o Hub mate todos os `next dev` filhos, sem vazar processos nem ports presas, para nao precisar `kill -9` manual depois.

## Tarefas tecnicas
1. [ ] Implementar `preview.stop(projectId)` em `lib/builder/preview.js`: SIGTERM no pid, aguarda 5s, SIGKILL se necessario, remove linha de `builder_preview_processes`.
2. [ ] Endpoint `POST /api/builder/projects/:id/preview/stop` no router.
3. [ ] Hook `process.on("exit"|"SIGINT"|"SIGTERM")` no boot do Hub: itera `builder_preview_processes`, mata todos os pids, limpa tabela.
4. [ ] Boot-time orphan check: na inicializacao do Hub, ler tabela; para cada pid, `process.kill(pid, 0)` para checar se vivo; se morto, remover linha; se vivo mas `next dev` nao atende, matar.
5. [ ] UI: botao "Parar preview"; trocar de projeto na lista chama stop do anterior antes de start do novo.

## Arquivos envolvidos
- `lib/builder/preview.js` - modificar (stop + cleanup)
- `lib/builder/router.js` - modificar (rota preview stop)
- `server.js` - modificar (signal handlers no boot/shutdown)
- `public/builder/app.js` - modificar (botao parar + troca de projeto)

## Definition of Done
- [ ] Clicar "Parar preview" mata o processo (verificavel via `ps aux | grep next`)
- [ ] `Ctrl+C` no Hub mata todos os previews ativos
- [ ] Reiniciar o Hub apos crash limpa zombies (boot orphan check funciona)
- [ ] Trocar de projeto B para C: B fica parado, C inicia
- [ ] Tabela `builder_preview_processes` reflete estado real
- [ ] Sem erros no console
- [ ] F5 preserva estado
