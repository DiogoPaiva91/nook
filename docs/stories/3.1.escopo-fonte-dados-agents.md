# Story 3.1: Definir fonte de dados dos agents BMad para a tela-piloto

## Contexto
Epico 3 (Piloto). Resolve item 3 da §10 da arquitetura: filesystem adapter vs sync para Postgres. Spike de 1h conforme §9 da arquitetura. Pre-requisito de 3.2 (construir a tela). Sem codigo do builder novo aqui — apenas decisao + adapter minimo.

## Descricao
Como Diogo, quero decidir como a tela de gerenciamento de agents BMad le e escreve dados (filesystem `bmad/agents/*.md` direto via Edge Function adapter, OU sync para tabela Postgres) para que a Story 3.2 saiba qual binding usar.

## Tarefas tecnicas
1. [ ] Spike timeboxed (1h): listar tradeoffs Filesystem-adapter vs Postgres-sync em `docs/stories/3.1-spike.md`. Criterios: latencia, complexidade, conflito com source-of-truth do BMad.
2. [ ] Decidir e documentar em ADR-8 dentro de `docs/architecture.md` (apos §7) — ou nota de decisao se nao for ADR formal.
3. [ ] Implementar adapter minimo: se filesystem -> Edge Function `app/api/agents/route.ts` no projeto-piloto que le `bmad/agents/*.md`; se Postgres -> tabela `agents` + script de seed/sync inicial dos arquivos `.md`.
4. [ ] Documentar caminho de escrita (criar/editar/deletar) coerente com a fonte escolhida. Garantir que o BMad real continua funcionando.
5. [ ] Garantir que a entidade `Agent` no `builder.json` do projeto-piloto reflete a decisao (campos: id, name, prompt, file_path se filesystem).
